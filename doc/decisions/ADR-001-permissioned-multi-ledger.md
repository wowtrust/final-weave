# ADR-001：许可型多账本

- 状态：Accepted
- 日期：2026-07-23
- 关联需求：FR-LEDGER-*、SAFE-010、FR-EPOCH-*

## 背景

FinalWeave 需要服务多个机构共同维护的业务账本，同时允许一个节点承载多个业务域。开放匿名验证者需要代币、Sybil 防御、质押和经济惩罚，会显著改变威胁模型。

## 决策

FinalWeave v1 采用许可型网络：

- 每个 Ledger 有独立 Genesis 和 LedgerID；
- 每个 epoch 有固定 validator set；
- validator 由组织证书和共识公钥识别；
- 成员变化通过治理交易并在 epoch 边界生效；
- 一个 Node Runtime 可承载多个 Ledger Runtime；
- 每个 Ledger 的共识、状态、网络 topic、存储和资源配额隔离；
- 跨账本 v1 使用证明驱动的异步消息。
- 各账本在同一 epoch 内固定使用 FinalDAG-C，不因其他账本负载或故障动态切换共识。

## 安全与信任影响

- Sybil 防御来自许可治理，不来自 PoW/PoS；
- 安全假设为每账本不超过 `f` 个 Byzantine validator；
- 组织 PKI、离线治理根和密钥管理进入系统信任边界；
- 成员发现不能改变 validator set。

## 正面结果

- 明确治理和责任主体；
- 确定性最终性适合企业业务；
- 多账本避免为每个业务部署完全独立软件栈；
- 可以按账本设置成员、状态机和保留策略。

## 代价与风险

- 不具备无许可公链的开放性；
- PKI 和治理流程可能形成组织层风险；
- 多账本共置会产生资源争抢；
- 跨账本不能天然共享最终性。

## 备选方案

- 无许可 PoS：v1 拒绝，范围和经济模型过大；
- 单一全局账本：拒绝，无法满足业务隔离；
- 每账本独立进程：可作为部署选择，但不作为唯一架构。

## 验证方式

- 4/7/10 validator epoch 切换测试；
- 多账本资源隔离测试；
- validator 证书吊销测试；
- 跨账本 proof message 重放测试。
- 一个账本资源耗尽时另一个账本的 DAG/执行签名安全测试。
